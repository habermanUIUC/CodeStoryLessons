Coding Cautions

All notebooks have to run in the testing framework. It's important that the following rules are adhered to:

Only import libraries requested

Each lesson will be specific as to what Python libraries can be imported. These libraries will also be available on the testing machine (and no others).

Reading Data will Fail

Be careful of using open(filename, 'r'). The filename will not exist on the test machine (you will be able to read files that you write however).

If a lesson has an accompanying dataset, you will use the LessonUtil to read the data:

def test_reader():
    import LessonUtil as Util
    path = Util.path_for_data('data.txt')
    with open(path, 'r') as fd:
       data = fd.read()
       print(data)
       
test_reader()

Encapsulate everything into a function or a class

It's a good habit and practice to always write functions (and classes).
Even the code that you write to test your own functions should be in a function:

def my_fancy_function(data):
    return [x for x in range(0, len(data))]

def test_it():
   data = open(some_file_name, 'r').read()
   answer = my_fancy_function(data)
   assert len(answer) == len(data), "Bad Result"

test_it()

The testing framework will not run any function calls that are at the module level (e.g. test_it() in the above example).

Writing Data Files

When you write to the file system

A few things to note:

  1. The file (e.g. test.data) is not permanent, it will disappear
  2. The file is saved at the top level (take a look at the files)

Remove the magic

Since notebooks allow displaying HTML (via %%), interacting with the shell (via ! commands), and other magic commands (via %), it's important that any code cells that have these 'features' need to be completely commented out. This will not be an issue for any lessons, but just a word of caution if you use these features, your code will no longer run in a pure python environment.
